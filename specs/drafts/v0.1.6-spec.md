# Planck v0.1.6 — Release Spec

## Overview

Three provider types, new config schema, SetupModal overhaul, simplified Docker
stack, and idempotent install scripts. Breaking config change — no backward
compatibility required.

---

## Phase 1 — Simplified provider model (`planck_ai`)

Drop `:custom_openai` entirely. Public provider atoms become:
`:anthropic | :openai | :google`.

### Provider routing

| Condition | req_llm provider |
|---|---|
| `:openai`, no `base_url` | `:openai` |
| `:openai`, `base_url` set | `:custom_openai` (internal) |
| `:anthropic` | `:anthropic` |
| `:google` | `:google` |

The `:custom_openai` / `:openai` distinction in req_llm becomes a private adapter
concern — never exposed in the public `planck_ai` API.

### API key resolution

- `has_api_key: false` → no key resolved (Ollama, llama.cpp)
- `identifier: "NVIDIA"` → `NVIDIA_API_KEY`
- no identifier → `OPENAI_API_KEY`

### Catalog bypass

Currently triggered by `:custom_openai` atom — change to trigger on
`base_url != nil` instead.

### Module changes

- `Planck.AI.Model` — `provider` field: `:anthropic | :openai | :google` only
- `Planck.AI.Models.CustomOpenAI` — merged into `Planck.AI.Models.OpenAI` or
  made internal
- `Planck.AI.Adapter` — routing logic updated; `identifier` defaults to `"OPENAI"`
  when nil on `:openai` provider
- `Planck.AI.Config` — drops `"custom_openai"` as valid provider string,
  maps `"openai"` to `:openai`
- Docs, specs, README updated throughout

## Phase 2 — New config schema (`planck_headless`)

Breaking change to `config.json`. No migration path needed.

### New shape

```json
{
  "default_provider": "anthropic",
  "default_model":    "sonnet",
  "sessions_dir":     ".planck/sessions",
  "skills_dirs":      [".planck/skills", "~/.planck/skills"],
  "teams_dirs":       [".planck/teams", "~/.planck/teams"],
  "sidecar":          ".planck/sidecar",
  "locale":           "en",
  "providers": {
    "anthropic":    { "type": "anthropic" },
    "openai":       { "type": "openai" },
    "google":       { "type": "google" },
    "nvidia": {
      "type":       "openai",
      "base_url":   "https://integrate.api.nvidia.com/v1",
      "identifier": "NVIDIA"
    },
    "groq": {
      "type":       "openai",
      "base_url":   "https://api.groq.com/openai/v1",
      "identifier": "GROQ"
    },
    "local-ollama": {
      "type":        "openai",
      "base_url":    "http://localhost:11434",
      "has_api_key": false
    }
  },
  "models": [
    {
      "id":       "sonnet",
      "model":    "claude-sonnet-4-6",
      "provider": "anthropic",
      "default":  true
    },
    {
      "id":       "llama70b",
      "model":    "meta/llama-3.3-70b-instruct",
      "provider": "nvidia",
      "params":   { "temperature": 0.6, "receive_timeout": 600000 }
    }
  ]
}
```

### Provider entry fields

| Field | Required | Description |
|---|---|---|
| `type` | yes | `"anthropic"`, `"openai"`, or `"google"` |
| `base_url` | no | Override base URL; triggers OpenAI-compatible routing |
| `identifier` | no | Uppercase tag for env var derivation (`NVIDIA` → `NVIDIA_API_KEY`). Defaults to `"OPENAI"` |
| `has_api_key` | no | `false` skips API key resolution. Default: `true` |

### Model entry fields

| Field | Required | Description |
|---|---|---|
| `id` | yes | User-defined alias used throughout the UI |
| `model` | yes | Provider model identifier (e.g. `"claude-sonnet-4-6"`) |
| `provider` | yes | Key referencing an entry in `providers` |
| `default` | no | Boolean; first model is default if none set |
| `params` | no | Inference parameters (`temperature`, `max_tokens`, etc.) |

### Code changes

- `Planck.Headless.Config` — new schema validation for `providers` map and
  simplified `models` list
- `JsonBinding` — handles `providers` key alongside `models`
- `ResourceStore` — resolves `provider key → provider entry + model entry →
  Planck.AI.Model` struct at load time
- `configure_provider/1` — new function (split from `configure_model/1`)
- `configure_model/1` — simplified; references provider by key
- `api_key_env_var/1` — updated for new provider shape

## Phase 3 — SetupModal / config UI overhaul (`planck_cli`)

### First-time onboarding (no models configured)

Shown automatically when `providers` and `models` are both empty.

**Step 1 — Add a provider**

- Provider type picker: Anthropic / OpenAI / Google / OpenAI-compatible
- API key field (hidden for `has_api_key: false` providers)
- For OpenAI-compatible: dropdown of popular presets:
  - NVIDIA NIM (base URL + identifier pre-filled)
  - Groq (base URL + identifier pre-filled)
  - Ollama (base URL pre-filled, no API key)
  - llama.cpp (base URL pre-filled, no API key)
  - Other (blank fields)
- "Get started for free" section with links to API key pages:
  - NVIDIA NIM — Llama 3.3 70B, Nemotron
  - Groq — Fast Llama/Gemma models
  - Google AI Studio — Gemini 2.5 Flash

**Step 2 — Add a model**

- Model picker: dropdown from provider catalog (cloud) or text input (custom)
- Alias field (pre-filled with model id)
- Set as default checkbox
- Parameters section:
  - Anthropic / OpenAI / Google: hidden by default, "Advanced" toggle reveals them
  - OpenAI-compatible: shown; pre-filled with good defaults for known providers
    (NVIDIA → `temperature: 0.6, receive_timeout: 600000`; Groq → `temperature: 0.5`)

### Config button (returning users)

Two options:
- **Configure a provider** — same as onboarding step 1
- **Configure a model** — pick existing provider, pick model, set alias/params/default

### Model dropdowns throughout the app

Chat overlays and orchestrator model selection show only configured models
(not the full catalog).

## Phase 5 — Idempotent install scripts

Both `install_docker.{sh,ps1}` and `install.{sh,ps1}` must be safely re-runnable
(running again = update to latest version, workspace preserved).

| Resource | Behavior |
|---|---|
| `compose.yml` | Always re-download (infra, not user data) |
| `~/planck/.env` | Never overwrite; add missing keys only |
| `workspace/` | Never touched (bind mount) |
| Model file | Removed — no longer downloaded |
| Docker images | Always pull latest |
| `setup` container | Always run (already idempotent) |
| CLI binary | Always replace; never touch `~/.planck/` or `.planck/` |

`.env` migration pattern:
```sh
add_if_missing() {
  grep -q "^$1=" "$ENV_FILE" || echo "$1=$2" >> "$ENV_FILE"
}
```

Note: `~/planck/.env` (Docker Compose vars) is separate from
`workspace/.planck/.env` (Planck app config / API keys) — the installer
never touches the latter.

## Phase 6 — Centralized version + CHANGELOG

### Single source of truth for the version

All five `mix.exs` files currently have an independent `@version "x.y.z"`.
Replace with a root `VERSION` file (plain text, one line) read at compile time:

```
VERSION          ← repo root, e.g. "0.1.6"
```

Each `mix.exs` reads it using `__DIR__` so the path is resolved relative to
the file, not the shell working directory:

```elixir
# planck_ai/mix.exs, planck_agent/mix.exs, planck_headless/mix.exs, planck_cli/mix.exs
@version File.read!(Path.join(__DIR__, "../VERSION")) |> String.trim()

# planck_docker/sidecar/mix.exs (one level deeper)
@version File.read!(Path.join(__DIR__, "../../VERSION")) |> String.trim()
```

Install scripts (`install_docker.{sh,ps1}`, `install.{sh,ps1}`) keep a
hardcoded `VERSION=` variable but it is updated as part of the release
process alongside the `VERSION` file.

### CHANGELOG structure

Each package keeps its own `CHANGELOG.md` (changes differ per package).
`planck_docker/CHANGELOG.md` covers Docker + install script changes.

For v0.1.6, add an `## Unreleased` section to each CHANGELOG now, then
rename it to `## v0.1.6` at release time. This avoids a last-minute
multi-file edit during the release commit.

### Files with hardcoded version (must update each release)

| File | What to update |
|---|---|
| `VERSION` | Version string itself |
| `planck_ai/mix.exs` | Reads `VERSION` after Phase 6 — no manual edit needed |
| `planck_agent/mix.exs` | Same |
| `planck_headless/mix.exs` | Same |
| `planck_cli/mix.exs` | Same |
| `planck_docker/sidecar/mix.exs` | Same |
| `planck_cli/lib/planck/web/live/status_bar.html.heex` | Use `Application.spec(:planck_cli, :vsn)` — automatic after Phase 6 |
| `planck_cli/lib/planck/web/api/spec.ex` | Same |
| `planck_docker/compose.yml` | Image tags: `ghcr.io/alexdesousa/planck:X.Y.Z` (2 lines) |
| `docs/install_docker.sh` | `VERSION="X.Y.Z"` |
| `docs/install_docker.ps1` | `$Version = "X.Y.Z"` |
| `docs/install.sh` | `VERSION="X.Y.Z"` (if present) |
| `docs/install.ps1` | `$Version = "X.Y.Z"` (if present) |

`status_bar.html.heex` and `api/spec.ex` currently have `"0.1.0"` hardcoded.
Replace with `Application.spec(:planck_cli, :vsn)`:

```heex
<%# status_bar.html.heex %>
<span class="opacity-50">v<%= Application.spec(:planck_cli, :vsn) %></span>
```

```elixir
# api/spec.ex
version: to_string(Application.spec(:planck_cli, :vsn)),
```

After Phase 6, only `compose.yml` and the 4 install scripts require manual
edits per release; all `mix.exs` files and the status bar read `VERSION`
automatically.

### Release checklist (replaces ad-hoc version bumping)

1. Update `VERSION` to the new version string.
2. Update image tags in `planck_docker/compose.yml`.
3. Update `VERSION=` in `docs/install_docker.{sh,ps1}` and `docs/install.{sh,ps1}`.
4. Rename `## Unreleased` → `## vX.Y.Z` in all CHANGELOG files.
5. Commit: `:bookmark: vX.Y.Z`
6. Tag: `git tag vX.Y.Z`
